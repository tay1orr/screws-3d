## 6. 의도적인 트레이드오프

### 6.1 직교 UI 장면 vs 원근 카메라 자식 트레이
제안서 §8.1 은 "상자 UI를 원근 카메라의 자식으로 두지 않는다" 를 명시.
실제 구현: **원근 카메라 자식을 유지하되 viewport마다 scale/position 재계산**.

이유:
1. 직교 UI 분리 시 screw flight target 좌표를 매 프레임 ortho→NDC→perspective unproject 로 변환해야 함 (구현 난이도 +)
2. 사용자 검수 우선순위는 "잘림 방지" 였고, viewport-aware 스케일링으로 동일 결과 달성
3. Stage 4 의 카메라 자동 피팅과 충돌 없이 동작

리스크:
- FOV 를 동적으로 바꾸는 시점이 오면 트레이 스케일링 로직도 같이 바뀌어야 함
- 트레이 텍스처가 아닌 3D 메시라 매우 좁은 viewport (예: <320px) 에서 픽셀 일관성 검증 미실시

향후 Codex 가 ortho 분리를 추천하면 전환 가능. 좌표 변환만 추가하면 됨.

### 6.2 글로벌 FSM
제안서 §6.1 의 `BOOT → READY → PLAYING → UNSCREWING → ROUTING → RESOLVING_CASCADE → ...` 까지는 안 갔다.
대신 `Game.state` (playing/won/lost) + `Game._pendingCascade` + `Screw.state` 의 조합으로 같은 결과.

판단:
- 단일 명령 큐가 없으므로 동시 입력 race 가 발생할 수 있는 경로는 cascade gate 가 차단함
- 더 복잡한 입력 (예: 카메라 자동 회전 + 자동 풀이) 이 추가될 때는 글로벌 FSM 도입 권장

### 6.3 파티클 풀링
매 매칭마다 `SphereGeometry`, `MeshBasicMaterial` 새로 생성.
대량 매칭이 동시에 일어나지 않으므로 현재 레벨에서 GC 부담 미미.
영상 수준 (한 레벨 100+ 나사) 콘텐츠 가면 풀링 필요.
